## Sample 1: Redis Migration Failure

### Input

```
Change: Migrating user session storage from Redis single node to Redis cluster mode with 3 shards across 3 AZs.

Current State:
- Single Redis node in us-east-1a
- 10,000 req/sec peak traffic
- Session TTL: 30 minutes
- Application uses redis-py client

Proposed Change:
- Redis cluster with 3 shards
- One shard per AZ (us-east-1a, 1b, 1c)
- Same traffic patterns
- No application code changes planned
```

## TRIGGERING EVENT

At 14:23 UTC, the Redis cluster was successfully deployed and health checks passed. However, the application code still used single-node Redis connection logic. When the first write operation attempted to store a session key, Redis returned a "MOVED" error indicating the key belonged to a different shard.

**Root Cause:** The application's Redis client (redis-py) was configured for single-node mode and did not understand cluster topology. Instead of following the MOVED redirect, it treated this as a connection error and initiated exponential backoff retries.

**Technical Details:**
- Error: `MOVED 3999 10.0.1.45:6379`
- Client behavior: Interpreted as connection failure
- Retry logic: 3 attempts per request with 100ms, 200ms, 400ms delays
- Result: 3x traffic amplification on every session write

### Dependency 1: Redis Client Cluster Awareness
**Assumption:** The redis-py client would automatically handle cluster mode  
**Reality:** Client required explicit cluster mode configuration (`RedisCluster` class instead of `Redis` class)  
**Impact:** All write operations failed with MOVED errors, triggering retry storms

### Dependency 2: Load Balancer Health Checks
**Assumption:** Health checks validated Redis connectivity  
**Reality:** Health checks only tested TCP connection to Redis, not actual read/write operations  
**Impact:** Instances remained "healthy" despite being unable to write sessions

### Dependency 3: Session Middleware Fallback
**Assumption:** Session middleware had graceful degradation  
**Reality:** Middleware had no fallback mechanism; failed session writes caused 500 errors  
**Impact:** All authenticated requests failed immediately

### Dependency 4: Auto-Scaling Triggers
**Assumption:** Auto-scaling based on CPU would help  
**Reality:** CPU spiked due to retry logic, triggering scale-out of broken instances  
**Impact:** More instances amplified the problem, overwhelming Redis cluster

### Decision Point 2: Canary Deployment

**What Happened:** Change was deployed to all instances simultaneously  
**What Could Have Been Done:** Deploy to 5% of instances first, monitor for 30 minutes  
**Outcome if Done:** Would have detected errors affecting only 5% of traffic, easy rollback

### Decision Point 3: Application Code Update
**When:** Before infrastructure change  
**What Happened:** Infrastructure changed first, assuming application would adapt  
**What Could Have Been Done:** Update application to use `RedisCluster` client first, verify compatibility  
**Outcome if Done:** Application would have been cluster-ready before infrastructure change

### Decision Point 4: Health Check Enhancement
**When:** During change planning  
**What Happened:** Health checks only tested TCP connectivity  
**What Could Have Been Done:** Add health check that performs actual session write/read operation  
**Outcome if Done:** Load balancer would have immediately detected broken instances

### Decision Point 5: Circuit Breaker Implementation
**When:** T+3m (when errors started)  
**What Happened:** Application continued retrying indefinitely  
**What Could Have Been Done:** Implement circuit breaker to fail fast after 3 attempts  
**Outcome if Done:** Would have prevented retry storm and traffic amplification

**When:** T+8m (when auto-scaling triggered)

**Fragility:** High - False positive health checks masked the problem

**Fragility:** Critical - No graceful degradation path

## PREVENTIVE ACTIONS

### High Priority

1. **Update Application to Cluster-Aware Client** - Modify code to use `RedisCluster` instead of `Redis` class - **Effort: Low**
   - Change 1 line of code in Redis connection initialization
   - Add cluster node discovery configuration
   - Test in development environment first

2. **Deploy Cluster in Shadow Mode** - Run cluster alongside single-node for testing - **Effort: Medium**
   - Configure application to write to both Redis instances
   - Compare results for consistency
   - Monitor for 1 week before cutover

3. **Enhance Health Checks** - Add actual session write/read to health check endpoint - **Effort: Low**
   - Create `/health/redis` endpoint that performs real operation
   - Configure load balancer to use new endpoint
   - Set appropriate timeout (2 seconds)

4. **Implement Circuit Breaker** - Add circuit breaker pattern for Redis operations - **Effort: Medium**
   - Use library like `pybreaker` or `circuitbreaker`
   - Configure: 5 failures in 10 seconds opens circuit
   - Half-open state after 30 seconds

### Medium Priority

5. **Gradual Traffic Migration** - Implement feature flag for Redis cluster usage - **Effort: Medium**
   - Add feature flag: `use_redis_cluster`
   - Start with 1% of traffic
   - Increase gradually: 1% → 5% → 25% → 50% → 100%

6. **Add Monitoring for MOVED Errors** - Create alert for Redis MOVED responses - **Effort: Low**
   - Add metric: `redis.moved_errors`
   - Alert if > 0 in 5-minute window
   - Include in deployment runbook

7. **Pause Auto-Scaling During Deployments** - Disable auto-scaling for 30 minutes post-deployment - **Effort: Low**
   - Add deployment hook to suspend auto-scaling
   - Re-enable after manual verification
   - Document in deployment process

8. **Create Rollback Runbook** - Document exact steps to rollback Redis changes - **Effort: Low**
   - Include commands, timing, verification steps
   - Practice rollback in staging
   - Keep runbook accessible during deployments

### Low Priority

9. **Implement Session Fallback** - Add in-memory session fallback for Redis failures - **Effort: High**
   - Use local cache as temporary session store
   - Sync to Redis when available
   - Accept eventual consistency

10. **Add Chaos Engineering Tests** - Regularly test Redis failure scenarios - **Effort: Medium**
    - Use tools like Chaos Monkey
    - Test: node failures, network partitions, high latency
    - Run monthly in staging

## Sample 2: Database Migration Failure

### Input

```
Change: Migrating from RDS MySQL single-AZ to Multi-AZ deployment

Current State:
- RDS MySQL 8.0 in us-east-1a
- db.r5.xlarge instance
- 500GB storage
- 2,000 connections peak
- Read replicas in us-east-1b and us-east-1c

Proposed Change:
- Enable Multi-AZ
- Automatic failover to us-east-1b
- Same instance type
- Synchronous replication
```

### Key Findings

**Assumed Outcome:** 15-minute outage during failover test

**Triggering Event:** Connection pool exhaustion during Multi-AZ synchronization

**Hidden Dependencies:**
- Application connection timeout too short (5s)
- No connection retry logic
- Monitoring only checked primary, not replica lag

**Preventive Actions:**
1. Increase connection timeout to 30s
2. Implement exponential backoff retry
3. Test failover in staging first
4. Monitor replication lag before enabling Multi-AZ

## Sample 3: Kubernetes Upgrade Failure

### Input

```
Change: Upgrading EKS cluster from 1.27 to 1.28

Current State:
- EKS 1.27
- 50 nodes across 3 AZs
- 200 pods running
- Ingress using ALB controller
- Service mesh: Istio 1.18

Proposed Change:
- In-place upgrade to EKS 1.28
- Rolling node replacement
- No application changes
```

### Key Findings

**Assumed Outcome:** Complete service outage for 2 hours

**Triggering Event:** Istio incompatibility with Kubernetes 1.28

**Hidden Dependencies:**
- Istio 1.18 not compatible with K8s 1.28
- Custom admission webhooks using deprecated API
- PodDisruptionBudgets too restrictive for rolling upgrade

**Preventive Actions:**
1. Upgrade Istio to 1.19 first
2. Update admission webhooks to v1 API
3. Relax PodDisruptionBudgets temporarily
4. Test upgrade in non-production cluster first
